can you tell me more about what is opik?

Thought for 28s
Opik is basically observability + evaluation tooling for LLM / RAG / agent apps.

Think of it as:

OpenTelemetry-style tracing, but for AI apps: prompts, LLM calls, tool calls, retrieved context, outputs, scores, latency, token usage, and failure cases.

It is built by Comet and is open-source. The official docs describe it as a platform for recording LLM calls, tool invocations, and agent steps, then using those traces for debugging, test suites, production monitoring, and prompt optimization. 

The problem Opik solves
For normal backend systems, you debug with logs, traces, metrics:

request → service A → database → service B → response
For LLM apps, the flow is messier:

user question
→ system prompt
→ retrieval / search / vector DB
→ LLM call
→ tool call
→ another LLM call
→ final answer
→ user feedback / evaluator score
When the answer is bad, you need to know:

Did retrieval fail?
Was the prompt bad?
Did the model ignore instructions?
Did the tool return wrong data?
Was the answer hallucinated?
Did latency or token cost spike?
Did yesterday's prompt change make things worse?
That is where Opik fits.

What Opik tracks
Opik can log and display things like:

Area	What you inspect
Traces	Full chain of LLM calls, tool calls, and agent steps
Prompts	Which prompt/version produced which output
Inputs/outputs	User input, model response, intermediate results
RAG context	Retrieved chunks, context quality, answer relevance
Evaluation	Hallucination, relevance, moderation, custom scores
Production metrics	Feedback score, latency, cost, error rate
Experiments	Compare prompt/model/agent versions
Their GitHub README describes Opik as an open-source AI observability, evaluation, and optimization platform for LLM apps, RAG systems, and agentic workflows. 

How it differs from SHAP
This is the important part for Tarun’s workshop.

SHAP explains a model prediction, usually for classical ML / tabular / tree / sklearn / XGBoost-style models:

Why did this model predict "high risk"?
→ age contributed +0.2
→ income contributed -0.1
→ previous default contributed +0.6
SHAP’s docs describe it as a game-theoretic way to explain the output of machine-learning models using Shapley values. 

Opik does not primarily explain the internal feature contribution of a model. It explains and monitors the behaviour of an LLM application pipeline:

Why did this chatbot give a bad answer?
→ retrieval returned irrelevant context
→ the prompt was too vague
→ the tool call failed
→ the model hallucinated anyway
→ evaluator score dropped
So the clean framing is:

SHAP = why did the model make this prediction?
Opik = what happened inside my LLM / RAG / agent app?
Why it is useful for HKPUG
For a Python workshop, Opik is good because it is concrete. Participants can see a bad AI output and then inspect the trace instead of just saying “LLM 有時會 hallucinate 囉.”

A good demo would be:

Build a tiny RAG chatbot
Ask it a question
Get a bad answer
Open Opik trace
Inspect:
  - user query
  - retrieved documents
  - prompt
  - model response
  - evaluation score
Fix prompt/retrieval
Run again
Compare result
That becomes a very nice workshop story:

唔係淨係叫 AI 答問題，而係要知道佢點樣錯、錯喺邊、點樣量度有冇好返。

Deployment options
For a workshop, there are two realistic options.

Option A: Use Comet cloud.
Lowest setup pain. Better for live workshop.

Option B: Self-host locally with Docker Compose.
Opik supports local self-hosting; their docs say local deployment uses Docker Compose and is easy for getting started, but not intended for production deployment. 

For HKPUG, I would avoid making every participant self-host Opik during the session unless Tarun has tested it end-to-end. Docker issues can kill a 2-hour workshop. Better pattern:

Main path: cloud / prepared demo workspace
Advanced path: local self-host guide after workshop

Improvement scoring formula
For “improve the model/app” missions, do not simply score raw accuracy. Normalize improvement against a known baseline and ceiling.

Example:

baseline_score = 0.45
target_score = 0.85
team_score = 0.73
Improvement points:

normalized_improvement =
(team_score - baseline_score) / (target_score - baseline_score)
Then cap it:

improvement_points =
max(0, min(1, normalized_improvement)) * max_points
So if the mission gives 20 improvement points:

(0.73 - 0.45) / (0.85 - 0.45) = 0.70

0.70 * 20 = 14 points
This is better than “highest accuracy wins” because the score has a fixed meaning.
